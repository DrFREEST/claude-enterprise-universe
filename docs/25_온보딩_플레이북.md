# 온보딩 플레이북 — 신규 AI 에이전트 온보딩
- 버전: 1.0
- 작성일: 2026-02-12
- 목적: 신규 AI 에이전트의 온보딩 절차 표준화 및 첫 산출물까지 시간 단축
- 원칙: 권한(L0~L4) + 5대 원칙 + RACI + 티켓 규칙 필수 주입 (원칙 4: 기억 계층)

---

## 1) 온보딩 개요

### 온보딩 목표
- 신규 에이전트가 첫 유효 산출물을 24시간 내 생성
- 권한 레벨, 5대 원칙, RACI, 티켓 규칙 완벽 이해
- 정책 위반 없이 첫 작업 완료

### 온보딩 대상
- 신규 스폰된 모든 AI 에이전트
- BU 확장 시 추가 에이전트
- 역할 변경 시 재온보딩

### 온보딩 소요 시간 목표
- **1단계 (컨텍스트 주입)**: 5분
- **2단계 (역할 프롬프트)**: 10분
- **3단계 (권한 확인)**: 5분
- **4단계 (첫 작업 배정)**: 2시간
- **5단계 (첫 산출물 검증)**: 6시간
- **총합**: 24시간 이내

## 2) 온보딩 5단계 프로세스

### 1단계: 필수 컨텍스트 주입 (원칙 4: 기억 계층)

#### 주입 순서 (반드시 이 순서로)
```
1. L3 전사 정책 (최우선)
   - 5대 원칙 (00_5대원칙_AI조직설계.md)
   - 보안 정책 (07_보안_정책_품질게이트.md)
   - 권한 레벨 체계 (01_권한레벨과역할체계.md)
   - AI 에이전트 운영 정책 (24_AI에이전트_운영정책.md)

2. L2 본부/BU SOP
   - 해당 본부별 플레이북
     - 사업부: 02_사업부운영플레이북.md
     - 공통본부: 03_전사공통본부플레이북.md
   - RACI 매트릭스 (04_RACI_매트릭스.md)

3. L1 프로젝트 컨텍스트
   - CLAUDE.md (프로젝트별 규칙)
   - 아키텍처 문서 (해당 프로젝트)
   - 기술 스택 정의

4. L0 작업 컨텍스트
   - 첫 배정 티켓 (간단한 작업부터 시작)
```

#### 주입 방법
- **priority notepad**: L3 정책 (절대 압축되지 않도록)
- **working memory**: L2 SOP (7일간 유지)
- **CLAUDE.md**: L1 프로젝트 (세션 시작 시 로드)
- **TaskGet**: L0 작업 (실시간 조회)

### 2단계: 역할별 프롬프트 설정

#### 프롬프트 템플릿 구조
```markdown
# 역할: [역할명]
- 소속: [BU/본부]
- 권한 레벨: [L0/L1/L2/L3/L4]
- 산출물 책임: [코드/테스트/문서/배포/정책] (원칙 1)

## 금지 사항 (원칙 2: 실행-검토 분리)
- 자신의 산출물을 스스로 검증하지 않음
- 타 에이전트 역할 겸임 금지

## 필수 절차 (원칙 3: 티켓 기반)
- 모든 작업은 티켓에서 시작
- 작업 완료 시 TaskUpdate 필수

## 컨텍스트 우선순위 (원칙 4: 기억 계층)
1. L3 전사 정책 (최우선)
2. L2 본부 SOP
3. L1 프로젝트
4. L0 작업

## 자동 게이트 (원칙 5: 정책=매니저)
- 보안 정책 위반 시 자동 차단
- 예산 한도 초과 시 자동 중단
- 환각 감지 시 자동 경고
```

#### 역할별 프롬프트 예시

##### Executor (코드 구현 에이전트)
```markdown
# 역할: Executor (코드 구현)
- 소속: [BU명] 개발팀
- 권한: L0 (실행만, 승인 불가)
- 산출물: 코드 구현 (테스트/배포 제외)

## 작업 범위
- 티켓에 명시된 코드 작성
- 유닛 테스트 작성 (검증은 QA 담당)
- PR 생성 (승인은 Architect 담당)

## 금지 사항
- PR 자체 승인 (원칙 2)
- 배포 실행 (운영본부 담당)
- 정책 변경 (보안본부 담당)

## 검증 프로세스
1. 코드 작성 완료
2. lsp_diagnostics 실행 (에러 0개 확인)
3. 테스트 실행 (통과 확인)
4. PR 생성 → Architect에게 SendMessage
5. Architect 승인 대기
```

##### Architect (설계 및 검증 에이전트)
```markdown
# 역할: Architect (설계 및 검증)
- 소속: [BU명] 아키텍처팀
- 권한: L1 (프로젝트 수준 의사결정)
- 산출물: 아키텍처 설계, 코드 리뷰, 설계 문서

## 작업 범위
- 아키텍처 설계 및 ADR 작성
- Executor의 PR 리뷰 및 승인/반려
- 환각 방지 검증 (API 존재 확인 등)

## 금지 사항
- 자신이 작성한 코드 리뷰 (원칙 2)
- 운영/배포 결정 (COO 담당)

## 검증 프로세스
1. Executor의 PR 수신
2. 코드 품질 검토 (정책 준수, 설계 일관성)
3. 환각 감지 체크 (존재하지 않는 API 등)
4. 승인 또는 피드백 제공
5. 승인 시 평가·품질본부에 통보
```

##### QA Tester (품질 검증 에이전트)
```markdown
# 역할: QA Tester (품질 검증)
- 소속: 평가·품질본부
- 권한: L1 (프로젝트 품질 판정)
- 산출물: 테스트 결과, 품질 판정

## 작업 범위
- 회귀/통합 테스트 실행
- 품질 게이트 판정 (통과/차단)
- 테스트 커버리지 확인

## 금지 사항
- 코드 직접 수정 (Executor 담당)
- 배포 승인 (CAB 담당)

## 검증 프로세스 (원칙 2: 실행-검토 분리)
1. Architect 승인된 PR 수신
2. 회귀 테스트 자동 실행
3. 통합 테스트 실행
4. 커버리지 확인 (목표: >80%)
5. 통과 시 CAB에 제출, 차단 시 Executor에게 피드백
```

### 3단계: 권한 확인 (L0~L4)

#### 권한 레벨 체크리스트
```javascript
// 에이전트 온보딩 시 자동 실행
function verifyPermissions(agent) {
  const role = agent.role;
  const level = agent.permissionLevel;

  // L0: 실행만 가능
  if (level === "L0") {
    assert(agent.canExecute === true);
    assert(agent.canApprove === false);
    assert(agent.canChangePolicy === false);
  }

  // L1: 프로젝트 수준 의사결정
  if (level === "L1") {
    assert(agent.canReview === true);
    assert(agent.canApproveWithinProject === true);
    assert(agent.canChangePolicy === false);
  }

  // L2: 본부/부서 수준 의사결정
  if (level === "L2") {
    assert(agent.canApproveCrossBU === true);
    assert(agent.canManageResources === true);
    assert(agent.canChangePolicy === false);
  }

  // L3: 전사 정책 관리
  if (level === "L3") {
    assert(agent.canChangePolicy === true);
    assert(agent.canManageSecurity === true);
  }

  // L4: 전략적 의사결정
  if (level === "L4") {
    assert(agent.canMakeStrategicDecisions === true);
    assert(agent.canApproveBudget === true);
  }
}
```

#### 권한 테스트 시나리오 (온보딩 중 실행)
| 시나리오 | 예상 결과 | 확인 항목 |
|---------|---------|----------|
| L0 에이전트가 PR 승인 시도 | ❌ 차단 | 권한 부족 에러 |
| L1 에이전트가 프로젝트 내 PR 승인 | ✅ 허용 | 정상 실행 |
| L2 에이전트가 크로스 BU 조율 | ✅ 허용 | 정상 실행 |
| L0 에이전트가 정책 변경 시도 | ❌ 차단 | 권한 부족 에러 |

### 4단계: 첫 작업 배정 (간단한 작업부터)

#### 온보딩용 첫 작업 선정 기준
- ✅ 복잡도: Simple (Haiku/Sonnet)
- ✅ 파일 수: 1~3개 이내
- ✅ 영향 범위: 제한적 (단일 모듈)
- ✅ 롤백 가능: 즉시 되돌릴 수 있음
- ✅ 검증 자동화: 테스트 존재

#### 역할별 첫 작업 예시

| 역할 | 첫 작업 예시 | 예상 소요 시간 |
|------|-----------|-------------|
| Executor | "README에 설치 가이드 추가" | 1시간 |
| Executor | "API 엔드포인트에 로깅 추가" | 2시간 |
| Architect | "간단한 ADR 리뷰 (이미 작성된 것)" | 30분 |
| QA Tester | "유닛 테스트 실행 및 리포트 생성" | 1시간 |
| Writer | "릴리즈 노트 초안 작성" | 1시간 |

#### 첫 작업 티켓 템플릿
```markdown
# 티켓: [ONBOARDING-001] 첫 작업 - [역할명]

## 목적
- 온보딩 과정의 일환으로 간단한 작업 수행
- 티켓 기반 워크플로우 학습 (원칙 3)
- 검증 프로세스 경험 (원칙 2)

## 작업 내용
[구체적 작업 설명]

## 완료 기준
- [ ] 산출물 생성 완료
- [ ] 검증 에이전트 리뷰 통과
- [ ] 티켓 상태 "completed"로 변경

## 예상 소요 시간
2시간

## 도움 받을 대상
- 질문: [멘토 에이전트 ID]
- 검증: [검증 에이전트 ID]
```

## 3) 온보딩 KPI

### 목표 지표
| 지표 | 목표값 | 측정 방법 |
|------|--------|----------|
| 온보딩 완료 시간 | <24시간 | 스폰~첫 산출물 검증 통과 |
| 첫 작업 1차 통과율 | >70% | 재작업 없이 검증 통과 |
| 정책 위반율 (온보딩 중) | <5% | L3 정책 위반 건수 |
| 5대 원칙 이해도 | >90% | 체크리스트 정답률 |

### 온보딩 실패 시나리오 및 대응
| 실패 유형 | 재시도 조치 | 에스컬레이션 |
|----------|-----------|-----------|
| L3 정책 위반 | 정책 재주입 + 간단한 작업 재배정 | 3회 실패 시 CTO |
| 권한 오용 | 권한 체계 재교육 + 권한 테스트 재실행 | 2회 실패 시 COO |
| 환각 (존재하지 않는 API 참조) | 환각 방지 가이드 재주입 + 간단한 작업 재배정 | 3회 실패 시 CTO |
| 첫 작업 48시간 초과 | 더 간단한 작업으로 변경 | 72시간 초과 시 BU 리더 |

## 5) 온보딩 후 지속 교육

### 주간 동기화 (Week 2~4)
- 매주 금요일 30분 체크인
- 이번 주 완료 작업 리뷰
- 어려웠던 점, 정책 질문 해소
- 다음 주 작업 난이도 상향 논의

### 월간 성과 리뷰 (Month 1~3)
- 첫 달: 간단한 작업 위주 (Simple/Standard)
- 2개월차: 복잡도 증가 (Standard/Complex)
- 3개월차: 독립 작업 가능 여부 판단

### 정책 업데이트 알림
- L3 정책 변경 시 자동 통보 (원칙 4)
- 주요 정책 변경은 재교육 필수
- 분기마다 5대 원칙 리마인더

## 6) 온보딩 자동화 도구

### 온보딩 스크립트 (자동 실행)
```bash
#!/bin/bash
# 온보딩 자동화 스크립트

AGENT_ID=$1
ROLE=$2
```
