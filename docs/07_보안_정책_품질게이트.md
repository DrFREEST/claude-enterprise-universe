# 보안 · 정책 · 품질 게이트 정의(Enterprise 기준)
- 작성일: 2026-02-12

---

## 1) 게이트 원칙
- 배포 전 필수: 품질 게이트(L1) + 보안/정책 게이트(L3)
- 미통과 시 운영본부는 배포를 승인하지 않는다(L2)

---

## 2) 품질 게이트(평가·품질본부)
- 통합/회귀 테스트, 성능 벤치(p95/p99), 정적 분석
- 산출물: 통과/차단 리포트, 해결 가이드
- 권한: L1(차단권)

---

## 3) 보안/정책 게이트(보안·정책·준법본부)
- Policy-as-Code: 승인 규칙/권한 규칙/금지 규칙/감사 규칙
- 체크: 비밀정보 탐지, 취약점 스캔, 권한/네트워크, 데이터 경계/보존
- 산출물: 정책 판정 리포트, 예외 기록
- 권한: L3

---

## 4) 긴급 예외(Sev0/1)
- 품질 축소: 평가·품질본부 판정
- 정책 예외: 보안본부 판정
- 배포 최종: 운영본부 승인
- 예외는 반드시 재발 방지 과제로 전환

---

## 5대 원칙 적용

이 문서는 다음 원칙을 반영합니다:

| 원칙 | 적용 방법 |
|------|----------|
| P2 실행-검토 분리 | 보안 정책 작성자(CISO 조직)와 보안 감사 실행자가 분리. 정책 예외 승인도 작성자가 하지 않음 |
| P5 정책=매니저 | 보안/품질 게이트가 매니저 역할. 위반 시 자동 차단, 수동 개입 없이 정책이 판정 |

### 검증 포인트
- [ ] 보안 정책 작성과 감사 실행이 분리되어 있는가?
- [ ] 게이트 통과/반려가 자동화되어 있는가?
- [ ] 정책 예외 처리에 시간 제한과 사후 정상화 의무가 있는가?
