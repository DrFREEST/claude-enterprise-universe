# 권한 레벨과 역할 체계 (Enterprise 기준)
- 작성일: 2026-02-12

---

## 1) 권한 레벨 정의(L0~L4)
- **L0 실행권**
  - 티켓/스펙 기반 작업 수행
  - 변경 제안(PR), 변경 요청(CR) 생성
- **L1 검토권**
  - 코드리뷰 승인/반려
  - 회귀/통합 테스트 “합격/불합격” 판정
  - 품질 게이트 차단 권고(권고 + 차단권 일부 포함 가능)
- **L2 배포권**
  - 프로덕션 배포 승인
  - 롤백/차단/긴급조치 실행
  - 릴리즈 캘린더/릴리즈 열차 운영
- **L3 정책권**
  - Policy-as-Code(승인 규칙/권한/금지) 편집 및 강제
  - 보안 스캔/비밀정보/데이터 경계/로그 보존 정책 변경
- **L4 경영권**
  - 로드맵/예산/채용·감축/조직개편/고충 최종결정
  - 대외 리스크·위기 대응 최종 승인

---

## 2) 역할군별 권한 기본 배치(표준)
| 역할군 | 기본 권한 |
|---|---|
| 사업부(기획/개발/도메인 QA) | L0~L1 |
| 평가·품질본부(회귀/성능/게이트) | L1(차단권 포함) |
| 운영본부(CAB/릴리즈/SRE) | L2 |
| 보안·정책·준법본부 | L3 |
| 경영위원회(CEO/CFO/CHRO 등) | L4 |

---

## 3) 임원 KPI(현실형)
| 임원 | 핵심 KPI(예시) |
|---|---|
| CEO | 전사 OKR 달성률, 중대사고(Sev0) 0건, 포트폴리오 성과 |
| COO | 전사 SLO 달성률, MTTR, 변경 실패율, 운영 사고 재발률 |
| CTO | 개발 리드타임, 플랫폼 재사용률, 실행 성공률, p95 지연 |
| CPO | 기능 채택률, 로드맵 달성률, 고객 TTV(Time-to-Value) |
| CISO | 정책 위반 0건, 보안 사고 0건, 감사 적합 100% |
| CFO | 비용 예측 정확도, 단가(토큰/컴퓨팅) 절감, 예산 준수 |
| CHRO | 충원 리드타임, 온보딩 완료율, 고충 해결 리드타임 |
| 감사실장(CEO 직속) | 감사 개선 완료율, 재발 방지율 |

## 4) 임원 권한(최종 책임자)
| 의사결정/권한 | 최종 책임 |
|---|---|
| 제품 로드맵(분기/연간) | CEO + CPO (전략기획실 협의) |
| 예산/비용(연간/분기) | CEO + CFO |
| 프로덕션 배포 체계(L2) | COO(운영본부) |
| 정책/보안 규칙(L3) | CISO(보안·정책·준법본부) |
| 채용/인원 확충 | CEO + CHRO + CFO |
| 고충/분쟁 최종 조정 | CHRO(조정) + 감사실(조사) |
| 대외 커뮤니케이션 | CEO + 비서실/홍보·대외협력실 |

---

## 5) 사업부와 중앙본부의 "경계선"
- 사업부는 빠르게 만들되, 배포/정책은 중앙에서 통제한다.
- 중앙본부는 게이트만 세우지 말고 템플릿/도구/가이드를 제공한다.
- 갈등은 사람 싸움이 아니라 RACI와 게이트 정의로 해결한다.

## 5대 원칙 적용

이 문서는 다음 원칙을 반영합니다:

| 원칙 | 적용 방법 |
|------|----------|
| P1 단일 산출물 책임 | 각 권한 레벨(L0~L4)이 고유한 산출물 유형만 담당. L0은 코드 구현, L1은 품질 판정, L2는 배포 승인, L3은 정책 편집, L4는 전략 결정 |
| P2 실행-검토 분리 | 권한 레벨 간 Maker-Checker 관계 내재. 코드 작성자(L0)가 품질 판정(L1)이나 배포 승인(L2)을 겸임 불가 |
| P5 정책=매니저 | 권한 체계 자체가 정책으로 강제됨. 권한 레벨 위반 시 자동 차단 |

### 검증 포인트
- [ ] 각 역할의 산출물 유형이 명확히 분리되어 있는가?
- [ ] Maker-Checker 관계가 권한 레벨로 보장되는가?
- [ ] 권한 정책이 자동 강제 메커니즘을 갖추고 있는가?
