# AI 엔터프라이즈 유니버스 조직 운영 문서 세트 (Enterprise 기준)
- 작성일: 2026-02-12
- 범위: AI 직원(에이전트)만으로 구성된 **엔터프라이즈 규모** 회사 운영을 가정
- 목적: “사업부 5개 + 전사 공통본부” 체계에서 **권한/책임(RACI)/회의체/변경관리(CAB)/릴리즈/장애/보안·정책·품질 게이트**를 현실적으로 운영 가능하게 정의

---

## 1) 권한 레벨 표준(L0~L4)
- **L0 실행권**: 작업 수행/변경 제안(PR/티켓/CR)
- **L1 검토권**: 리뷰/테스트 승인(품질 판정), 차단 권고
- **L2 배포권**: 프로덕션 배포 승인/롤백 실행, 긴급 조치
- **L3 정책권**: Policy-as-Code(보안/승인/금지 규칙) 편집·강제
- **L4 경영권**: 로드맵/예산/채용/조직/고충 최종 의사결정

> 원칙: **사업부는 L0~L1**, 운영본부는 L2, 보안·정책·준법본부는 L3, 경영위원회는 L4를 최종 소유합니다.

### 2-3. 사업부 5개(BU)
- **BU1 워크플로우 자동화 사업부**
  - 제품기획팀 / 워크플로우 엔진팀 / 자동화 커넥터팀 / 도메인 QA팀 / 고객 솔루션팀
- **BU2 프로덕션 빌더 사업부**
  - 제품기획팀 / 빌드 파이프라인팀 / 배포 전략팀 / 환경·구성관리팀 / 도메인 QA팀
- **BU3 엔터프라이즈 연동 사업부**
  - 엔터프라이즈 제품기획팀 / 인증·권한 연동팀 / 레거시·데이터 연동팀 / 보안심사 대응팀 / 도메인 QA팀
- **BU4 AI 운영·거버넌스 사업부**
  - 운영정책 제품기획팀 / 관제·알림 자동화팀 / 변경관리 자동화팀 / 감사·증적 자동화팀 / 운영 QA팀
- **BU5 에이전트·템플릿 마켓 사업부**
  - 마켓 제품기획팀 / 템플릿·레시피팀 / 검수·심사팀 / 개발자 생태계팀 / 마켓 QA팀

### 2-4. 전사 공통본부(Shared Services)
- 운영본부: 변경관리실(CAB) / 릴리즈관리팀 / 서비스운영팀(SRE)
- 플랫폼본부: 파이프라인팀 / 런타임팀 / 관측팀 / 비용최적화팀

- 평가·품질본부: 회귀테스트팀 / 성능·신뢰성팀 / 품질게이트 운영팀
- 개발자경험본부(DX): 표준템플릿·SDK팀 / 내부도구팀 / 기술문서팀
- 고객성공본부: 온보딩팀 / 기술지원팀 / 장애 커뮤니케이션팀

## 4) 가장 중요한 운영 포인트(요약)
1. **배포권(L2)과 정책권(L3)은 사업부에서 분리**한다.
2. **변경관리(CAB)**를 통해 “왜/무엇/영향/롤백”이 없는 배포를 막는다.
3. **품질게이트(회귀/성능) + 보안게이트(정책/취약점) 통과 없이는 배포 불가**.
4. **인사/고충/감사**는 조직이 커질수록 독립성이 중요하다(특히 AI 조직도 동일).

## 5) AI 조직 설계 5대 원칙

> 모든 문서/정책/프로세스가 준수해야 할 불변 규칙. 상세: `00_5대원칙_AI조직설계.md`

| # | 원칙 | 핵심 규칙 |
|---|------|----------|
| P1 | 단일 산출물 책임 | 한 에이전트는 하나의 산출물 타입만 책임 |
| P2 | 실행-검토 분리 | 만든 사람(Maker) ≠ 검토자(Checker) |
| P3 | 티켓 기반 실행 | 모든 작업은 티켓(Task)에서 시작 |
| P4 | 기억 계층 분리 | 전사 정책(L3) > 부서 SOP(L2) > 프로젝트(L1) > 작업(L0) |
| P5 | 정책=매니저 | 보안/코딩/테스트/릴리즈 규칙이 자동 강제 |
