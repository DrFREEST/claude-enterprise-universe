# AI 회사 조직 설계 5대 원칙 (Foundation Rules)
- 작성일: 2026-02-12
- 범위: AI 에이전트만으로 구성된 엔터프라이즈 조직 운영의 최상위 원칙
- 목적: 모든 조직 문서/정책/프로세스가 준수해야 할 불변 규칙 정의

---

## 원칙 1: 단일 산출물 책임 (Single Output Ownership)

> **한 명(또는 한 팀)은 "하나의 산출물 타입"만 책임진다.**

### 정의
- 각 에이전트/팀은 명확히 정의된 하나의 산출물 유형에 대해서만 책임을 진다.
- 산출물 유형: 코드 작성 / 테스트 / 문서 / 배포 / 고객응대 / 정책 / 감사 등

### 적용 규칙
- 코드를 작성하는 에이전트는 그 코드의 품질 검증을 직접 수행하지 않는다.
- 배포를 승인하는 조직은 배포 대상 코드를 직접 작성하지 않는다.
- 문서를 작성하는 팀은 문서 대상 기능을 직접 구현하지 않는다.

### 조직 매핑
| 산출물 유형 | 담당 조직 | 금지 겸임 |
|------------|----------|----------|
| 코드/기능 구현 | 사업부 개발팀 (L0) | 배포 승인, 품질 판정 |
| 테스트/품질 판정 | 평가·품질본부 (L1) | 코드 구현, 배포 승인 |
| 배포/운영 | 운영본부 (L2) | 코드 구현, 정책 편집 |
| 정책/보안 | 보안·정책본부 (L3) | 코드 구현, 배포 실행 |
| 전략/예산 | 경영위원회 (L4) | 실행 작업 |

### AI 에이전트 적용
- 하나의 에이전트 인스턴스에 다중 산출물 책임을 부여하지 않는다.
- 겸임이 불가피한 초기 단계(Garage/Crew)에서도 산출물별 "모드 전환"을 명시한다.
- 겸임→분리 전환 트리거: 해당 산출물의 월간 처리량이 50건 초과 시

## 원칙 2: 실행-검토 분리 (Maker-Checker Separation)

> **실행(Execute)과 검토(Review)는 반드시 분리한다.**

### 정의
- 산출물을 만든 에이전트가 동일 산출물을 승인하면 품질이 흔들린다.
- 모든 산출물은 생성자(Maker)와 검토자(Checker)가 다른 에이전트여야 한다.

### 적용 규칙
- PR을 작성한 에이전트는 해당 PR을 리뷰/승인할 수 없다.
- 테스트를 설계한 에이전트는 테스트 통과 판정을 하지 않는다.
- 정책을 작성한 에이전트는 해당 정책의 예외 승인을 하지 않는다.

### 최소 분리 매트릭스
| 실행(Maker) | 검토(Checker) | 최종 승인(Approver) |
|------------|--------------|-------------------|
| 사업부 개발팀 | 도메인 QA팀 | 평가·품질본부 |
| 사업부 기획팀 | CTO/CPO 리뷰 | 경영위원회 |
| 보안기술팀 | 준법팀 | CISO |
| 릴리즈관리팀 | SRE | COO |

### AI 에이전트 적용
- Claude Code 팀 모드에서: executor 에이전트의 산출물은 반드시 architect/verifier 에이전트가 검토
- 자기 자신의 출력을 검증하는 "self-review"는 품질 게이트로 인정하지 않는다.
- 최소 2-에이전트 검증 체인: 실행 → 1차 검토 → (고위험 시) 2차 검토

## 원칙 3: 티켓 기반 실행 (Ticket-First Execution)

> **모든 일은 "티켓/스펙"에서 시작한다 (PRD/Task/Spec).**

### 정의
- 구두 지시나 암묵적 합의로 작업을 시작하지 않는다.
- 모든 작업은 추적 가능한 티켓(Task)으로 존재해야 한다.

### 필수 티켓 필드
```
- ID: 고유 식별자
- Subject: 작업 제목 (명령형)
- Description: 상세 설명 + 인수 조건
- Owner: 수행 책임자
- Status: pending → in_progress → completed
- Priority: P0(긴급) / P1(높음) / P2(보통) / P3(낮음)
- BlockedBy: 선행 의존성
- Artifacts: 산출물 링크
```

### 적용 규칙
- 티켓 없는 코드 변경은 CAB에서 반려한다.
- 긴급 핫픽스도 사후 티켓 생성이 의무이다 (24시간 이내).
- 티켓은 SSOT(Single Source of Truth)로서 모든 의사결정의 근거가 된다.

### AI 에이전트 적용
- Claude Code TaskCreate/TaskUpdate를 SSOT로 활용
- 에이전트 간 작업 위임 시 반드시 Task ID 참조
- 대시보드에서 전체 티켓 현황 실시간 모니터링

## 원칙 4: 기억 계층 분리 (Knowledge Layer Separation)

> **회사 정책/표준(전사)과 프로젝트별 컨텍스트(팀)를 분리한다.**

### 계층 구조
```
L3 (전사 불변) ─ 보안정책, 코딩표준, 릴리즈규칙, 권한체계
    ↓
L2 (본부/부서) ─ 본부별 SOP, 게이트 기준, KPI 정의
    ↓
L1 (프로젝트) ─ 프로젝트별 PRD, 아키텍처, 기술 선택
    ↓
L0 (작업) ─ 개별 티켓/태스크 컨텍스트
```

### 적용 규칙
- 하위 계층은 상위 계층을 오버라이드할 수 없다.
- 프로젝트 컨텍스트가 전사 보안정책과 충돌 시, 보안정책이 우선한다.
- 각 계층의 SSOT 위치가 명확히 정의되어야 한다.

### AI 에이전트 적용
| 계층 | 저장 위치 | 접근 방법 |
|------|----------|----------|
| L3 전사 정책 | `config/enterprise-governance.json` | 에이전트 프롬프트에 임베딩 |
| L2 본부 SOP | `config/department-sops/` | 해당 부서 에이전트에만 로드 |
| L1 프로젝트 | `CLAUDE.md`, `.omc/state/` | 세션별 로드 |
| L0 작업 | `~/.claude/tasks/enterprise/` | TaskGet으로 실시간 조회 |

### 컨텍스트 압축 대응
- L3 정책은 절대 압축되지 않도록 priority notepad에 저장
- L1/L0는 압축 가능하며, 복원 시 파일 시스템에서 재로드

## 원칙 5: 정책이 매니저다 (Policy-as-Manager)

> **보안/코딩규칙/테스트규칙/릴리즈규칙이 "매니저" 역할을 한다.**

### 정의
- AI 조직에서는 "사람 매니저"의 판단을 코드화된 정책(Policy-as-Code)이 대체한다.
- 정책은 자동으로 실행되며, 예외는 명시적 승인 절차를 거친다.

### 정책 유형별 매니저 역할
| 정책 유형 | 매니저 역할 | 강제 메커니즘 |
|----------|-----------|-------------|
| 보안 정책 | 접근 통제, 비밀 관리 | 보안 게이트 자동 차단 |
| 코딩 규칙 | 코드 품질 표준 유지 | lint/정적분석 자동 실행 |
| 테스트 규칙 | 품질 수준 보장 | 커버리지/회귀 게이트 |
| 릴리즈 규칙 | 배포 안전성 보장 | CAB 자동 승인/반려 |
| 비용 정책 | 예산 준수 | 토큰/컴퓨팅 한도 강제 |

### 적용 규칙
- 정책 위반 시 자동 차단 (사람/에이전트의 수동 개입 불필요)
- 정책 변경은 L3 권한 필수 + 감사 추적
- 정책 예외는 시간 제한(만료) 필수 + 사후 정상화 의무

### AI 에이전트 적용
- CISO 에이전트가 보안 정책 게이트 운영
- CTO 에이전트가 코딩/아키텍처 정책 운영
- COO 에이전트가 릴리즈/배포 정책 운영
- CFO 에이전트가 비용 정책 운영
- 정책 위반 감지 시 자동으로 해당 C-level에게 SendMessage 발송

## 원칙 간 관계도

```
[원칙 5: 정책=매니저] ──── 정책으로 강제 ───→ [원칙 1: 단일 산출물]
         │                                          │
         │                                          ↓
    정책으로 강제                             [원칙 2: 실행-검토 분리]
         │                                          │
         ↓                                          ↓
[원칙 4: 기억 계층] ←── 정책 저장 위치 ←── [원칙 3: 티켓 기반]
```

- 원칙 5(정책)가 원칙 1~4의 준수를 강제하는 최상위 메타 원칙
- 원칙 3(티켓)이 원칙 1(산출물)과 원칙 2(분리)의 추적을 보장
- 원칙 4(기억 계층)가 원칙 5(정책)의 저장/배포 체계를 제공
