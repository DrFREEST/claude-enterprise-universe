# 변경관리(CAB) · 릴리즈 · 장애 대응 SOP (Enterprise 기준)
- 작성일: 2026-02-12

---

## 1) 변경요청(CR) 템플릿
- 변경 제목/ID, 목적(왜), 내용(무엇)
- 영향 범위(서비스/고객/데이터)
- 위험도(낮음/중간/높음)
- 검증 증적(회귀/성능/보안 결과)
- 배포 방법(카나리/점진/블루그린)
- 롤백 플랜(트리거/절차/책임자)
- 커뮤니케이션 필요 여부

---

## 2) CAB 승인 흐름(정기 변경)
1) 사업부 CR 제출(L0)
2) 평가·품질본부 품질 판정(L1)
3) 보안·정책·준법본부 정책 판정(L3)
4) 운영본부 승인/반려(L2)
5) 릴리즈관리팀 배포 실행(L2)
6) 서비스운영팀 모니터링/롤백(L2)

---

## 3) 릴리즈 열차(정기)
- T-3: 기능 동결
- T-2: 품질/보안 게이트 완료
- T-1: 문서/공지 준비
- T: 카나리→점진 확대→완료
- T+1: 운영 결과 리뷰

---

## 4) 장애 대응(인시던트)
- 심각도: Sev0/1/2/3
- 역할: 장애 지휘관(운영), 기술 대응(사업부), 품질/보안(게이트), 커뮤니케이션(고객성공/홍보)
- 단계: 탐지→완화→원인분석→복구→사후분석/재발방지

KPI: MTTR, 탐지 리드타임, 재발률, 공지 리드타임

---

## 5대 원칙 적용

이 문서는 다음 원칙을 반영합니다:

| 원칙 | 적용 방법 |
|------|----------|
| P2 실행-검토 분리 | 변경 요청자(Maker)와 CAB 승인자(Checker)가 분리. 코드 작성자가 배포를 승인하지 않음 |
| P3 티켓 기반 실행 | 모든 변경은 CR(Change Request) 티켓에서 시작. 긴급 핫픽스도 사후 24시간 내 티켓 생성 의무 |
| P5 정책=매니저 | 릴리즈 규칙/보안 게이트가 자동 강제. 게이트 미통과 시 배포 자동 차단 |

### 검증 포인트
- [ ] 변경 요청자와 승인자가 완전히 분리되어 있는가?
- [ ] 모든 변경이 CR 티켓으로 추적되는가 (긴급 핫픽스 포함)?
- [ ] 릴리즈 게이트가 Policy-as-Code로 자동 강제되는가?
